# 현장 톡방 경쟁 분석

> 작성일: 2026-04-09
> 목적: "현장이 돌아가는 단 하나의 톡방" MVP 설계 근거

---

## A. 카카오톡 오픈채팅

### 왜 사장님이 이걸 쓰는가
- **무료**, **100% 보급률** (반장도 자재상도 고객도 다 씀)
- 사진 전송이 즉시 가능
- 학습 비용 제로
- "다른 앱 깔아달라"고 말하기 어려움

### 진짜 고통 포인트
- 사진/파일 30일 만료 → 3개월 전 견적서 못 찾음
- 검색이 사실상 불가 (현장명으로 못 찾음)
- 공과 사 섞임 → 가족 카톡 사이에 현장 사진
- 단톡방 7개 → "아까 그거 어디 단톡이었지?"
- 읽음 표시만 있고 **확인/완료 표시 없음**
- 누가 뭘 담당하는지 구조화 불가

### 5개 질문
1. **차별점**: 보편성. 세상 모든 사람이 이미 쓰고 있음.
2. **훔쳐올 것**: 사진 전송의 즉시성. 톡방 입장 마찰 제로.
3. **안티패턴**: 구조 없는 대화. 모든 주제가 하나의 스트림에 섞임.
4. **외부인**: 구분 없음. 고객도 반장도 같은 톡방.
5. **파일**: 30일 만료. 아카이브 불가. 이게 가장 큰 고통.

## B. Slack

### 핵심 분석
- **채널 = 현장** 매핑이 가능 (#강남아파트, #판교빌라)
- 쓰레드로 주제별 대화 분리
- @멘션으로 특정인에게 알림
- 리마인더로 할 일 관리
- 채널 내 "파일" 탭으로 사진/문서 아카이브

### 왜 현장 사장님에게 안 맞는가
- **영어 UI** (한국어 있지만 어색)
- **가격** (Pro 월 $8.75/인 → 반장 10명이면 월 10만원+)
- **복잡함** (워크스페이스, 채널, 쓰레드 개념 학습)
- 반장/자재상에게 "Slack 깔아주세요" 요청 불가

### 5개 질문
1. **차별점**: 채널 구조 + 쓰레드 + 검색. 구조화된 비동기 소통.
2. **훔쳐올 것**: **채널=현장** 매핑. 대화가 항상 컨텍스트 안에서.
3. **안티패턴**: 사용자에게 "워크스페이스 개념"을 가르치려는 것. 학습 비용.
4. **외부인**: Slack Connect로 외부인 초대 가능하나 비쌈.
5. **파일**: 채널별 파일 탭. 무료 플랜은 90일 이력 제한.

## C. Front (가장 중요 벤치마크)

### 핵심 분석
- **공유 수신함(Shared Inbox)**: 팀 전체가 하나의 수신함을 공유
- 외부인(고객)에게는 이메일로 소통 → 내부에서는 코멘트로 논의
- **할당(Assignment)**: "이 고객은 김OO 담당" 지정
- SLA 타이머: "이 메시지에 2시간 내 답장" 추적
- 자동화 규칙: "견적 키워드 → 자동 할당"

### 현장 톡방 적용 방안
- **현장 = 공유 수신함**: 현장에 관한 모든 대화가 하나의 인박스
- **고객 메시지 ↔ 내부 코멘트 분리**: 고객에게 보이는 것 vs 반장/매니저끼리 논의
- **할당**: "이 현장은 박반장 담당" → 알림이 박반장에게만
- **이것이 카톡과의 결정적 차이**: 외부(고객)와 내부(반장)가 같은 톡방에 있지 않음

### 5개 질문
1. **차별점**: 외부/내부 커뮤니케이션의 분리 + 결합. 고객에게는 깔끔하게, 내부에서는 솔직하게.
2. **훔쳐올 것**: **내부 코멘트** 패턴. 고객 메시지 옆에 "이건 추가 비용이야" 내부 메모.
3. **안티패턴**: 이메일 기반이라 실시간성 부족. 현장은 실시간이 필요.
4. **외부인**: 이메일로 소통. 고객은 Front의 존재를 모름.
5. **파일**: 이메일 첨부파일 자동 아카이브. 대화 내 인라인 표시.

## D. Intercom Messenger

### 핵심 분석
- 웹사이트/앱에 **임베디드 메신저**
- 봇이 먼저 응대 → 사람으로 에스컬레이션
- 고객 프로필(이전 대화, 사용 데이터) 옆에 표시
- 고객은 "이 회사 채팅"으로 인식 (Intercom 브랜드 모름)

### 5개 질문
1. **차별점**: 고객 컨텍스트가 대화 옆에 실시간 표시. "이 고객은 프로 플랜, 3번째 문의."
2. **훔쳐올 것**: **고객 프로필 사이드바**. 대화하면서 "이 현장 공정률 65%, 미수금 500만원" 보기.
3. **안티패턴**: 봇 우선 응대. 인테리어 현장에서 봇은 오히려 신뢰 하락.
4. **외부인**: 고객이 웹/앱에서 채팅. 로그인 불필요(쿠키/이메일 기반).
5. **파일**: 채팅 내 파일 첨부. 고객별 대화 이력에 자동 아카이브.

## E. 하이웍스 / 잔디

### 왜 대부분 실패하는가
- **"우리도 Slack 만들겠다"** 접근 → 기능 카피지만 커뮤니티/생태계 없음
- 사용자가 "카톡이면 되는데 왜 이걸 깔아야 하지?"에 답 못 함
- 관리자(대표)가 강제해도 직원이 결국 카톡으로 돌아감

### 잔디가 살아남은 이유
- 무료 티어가 넉넉 → 진입 장벽 제거
- 심플한 UI (Slack 대비)
- 한국어 네이티브

### 하이웍스가 쓰이는 이유
- 그룹웨어 번들 (결재, 메일, 캘린더, 드라이브) → 메신저만의 가치가 아니라 **묶음 가치**
- "이미 쓰고 있는 그룹웨어에 메신저가 달려있으니 쓰는 것"

### 5개 질문
1. **차별점**: 잔디=심플, 하이웍스=번들. 독자적 가치보다 부수적 가치.
2. **훔쳐올 것**: **번들 전략**. 메신저 단독이 아니라, 현장관리에 톡방이 붙어있는 구조.
3. **안티패턴**: "Slack 대체재"로 포지셔닝. 카카오톡 대체재로도 포지셔닝하면 안 됨.
4. **외부인**: 대부분 미지원. 내부 커뮤니케이션 전용.
5. **파일**: 드라이브 연동 (하이웍스). 잔디는 채팅 내 파일 공유.

## F. 네이버웍스

### 중소기업이 쓰는 이유
- **네이버 ID 연동** → 별도 가입 불필요
- **무료** (300명까지)
- 캘린더, 드라이브, 설문 통합
- 한국 중소기업에 친숙한 네이버 브랜드

### 5개 질문
1. **차별점**: 네이버 ID. "깔아만 주세요"가 가능한 유일한 업무 메신저.
2. **훔쳐올 것**: **기존 ID 연동** 마찰 제거. 카카오톡 ID로 톡방 참여 가능하게?
3. **안티패턴**: "기능이 너무 많은 그룹웨어" 느낌. 현장 사장님에게 설문/캘린더는 과잉.
4. **외부인**: 외부 사용자 초대 가능 (게스트). 별도 가입 필요.
5. **파일**: 네이버 드라이브 연동. 용량 제한 있음.

## G. Linear Comments

### 핵심 분석
- **이슈 안에서 대화**: 댓글이 이슈(태스크)에 종속됨
- 코멘트가 곧 히스토리 → "이 이슈에 대해 누가 뭐라고 했는지" 추적
- 컨텍스트 결합형: 대화가 항상 **작업 맥락** 안에서 발생
- 이슈 상태 변경도 타임라인에 표시 → "누가 언제 '완료'로 바꿨는지"

### 현장 톡방 적용
- **공정(Phase) 안에서 대화**: "목공 공정" 안에서 "오늘 천장 실링팬 보강 완료" 코멘트
- 대화가 공정에 종속되므로, 나중에 "이 공정에서 무슨 일이 있었나" 추적 가능
- 상태 변경 자동 기록: "박반장이 '목공'을 '완료'로 변경" → 증거

### 5개 질문
1. **차별점**: 대화 ≠ 채팅. 대화 = 작업 기록. 코멘트가 곧 히스토리.
2. **훔쳐올 것**: **공정 안에서 대화** 패턴. 현장 톡방 ≠ 빈 채팅방. 현장 톡방 = 현장 데이터 위에 대화.
3. **안티패턴**: 텍스트 전용. 사진이 핵심인 현장에서는 사진 퍼스트 UX 필수.
4. **외부인**: 공유 이슈 URL로 외부인에게 상태 공유 가능 (읽기 전용).
5. **파일**: 코멘트 내 첨부. 이슈별로 관련 파일이 자동 그루핑.

## 결론: 인테리어코치 현장 톡방 설계 원칙 10개

### 원칙 1: 대화는 항상 현장(Site) 안에서 일어난다
Slack의 채널=현장 매핑 + Linear의 컨텍스트 결합. "어디서 뭘 얘기했더라?"를 없앤다.

### 원칙 2: 외부(고객)와 내부(반장)는 분리한다
Front 패턴. 고객에게 보이는 메시지 vs 내부 코멘트. 같은 톡방에 섞이지 않는다.

### 원칙 3: 사진이 1등 시민이다
카톡과 달리 사진이 만료되지 않고, 공정별로 자동 분류된다. 사진 업로드 = 작업 기록.

### 원칙 4: 카톡 링크로 참여할 수 있어야 한다
반장/자재상에게 "앱 깔아주세요"는 불가능. 카톡 알림톡 → 웹 링크로 참여. 네이버웍스의 마찰 제거 교훈.

### 원칙 5: 대화가 곧 기록이다
Linear 패턴. "박반장: 목공 완료" = 공정 상태 업데이트. 채팅과 작업 관리가 하나.

### 원칙 6: 메신저가 아니라 현장 관리에 톡방이 붙어있는 것이다
하이웍스의 번들 교훈. 독립 메신저로 포지셔닝하면 카톡을 못 이긴다. 현장 데이터 위에 대화를 얹는다.

### 원칙 7: 읽음 확인이 아니라 "확인/완료" 확인이다
카톡의 한계. "읽었다"와 "알겠다"는 다르다. 반장이 일정 변경을 "확인" 누르면 그게 기록.

### 원칙 8: 검색은 현장명 × 공종 × 날짜로 가능해야 한다
카톡의 최대 고통. "지난 달 강남 아파트 타일 관련 대화" 검색이 3초 안에 되어야 한다.

### 원칙 9: 알림은 역할별로 차등한다
모든 메시지가 모든 사람에게 가면 Slack의 알림 피로. 반장에게는 자기 현장만, 사장님에게는 미수금/일정 변경만.

### 원칙 10: 30일 후에도 사진이 살아있어야 한다
카톡 최대 고통의 정반대. 1년 전 시공 사진도 현장 안에서 바로 열림. 이것만으로 전환 이유.
